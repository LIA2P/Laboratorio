Exploración de los datos¶

Siempre es recomendable inspeccionar los datos con los que vamos a trabajar. Al utilizar im√°genes esto es especialmente sencillo, ya que solamente tenemos que representarlas. Las dos cosas en las que deber√≠amos fijarnos son:

  • Las dimensiones de las im√°genes.
  • El rango de valores de los p√≠xeles.
  • El formato de las etiquetas.

Para comprobar las dimensiones podemos utilizar el atributo .shape, mientras que para observar el rango de valores podemos realizar un histograma o representar la imagen junto a su mapa de color.

Las capas convolucionales que utilizaremos requieren que las im√°genes de entrada tengan unas dimensiones determinadas: $(alto, ancho, canales)$. Adem√°s, si trabajamos con im√°genes muy grandes puede ser recomendable reducir su tama√±o para acelerar los c√°lculos. En este caso, las dimensiones de las im√°genes que hemos importado son $(28,28)$. Esto sucede porque las imagenes de este conjunto de datos est√°n en blanco y negro, por lo que solamente tienen un canal y han decidido no incluirlo. Podemos a√±adir este canal de color utilizando la funci√≥n np.expand_dims(), que permite a√±adir dimensiones "fantasma" a cualquier np.array:

Separación en entrenamiento-validación-test¶

Para poder ajustar algunos hiperpar√°metros del modelo es conveniente utilizar tambi√©n un conjunto de validaci√≥n, as√≠ que tomaremos 10000 muestras del conjunto de entrenamiento y las pasaremos a un nuevo conjunto de validaci√≥n. De esta forma obtenemos los tres conjuntos con los que vamos a trabajar:

  • Entrenamiento: 50000 muestras.
  • Validaci√≥n: 10000 muestras.
  • Test: 10000 muestras.

Hemos elegido esta proporci√≥n para que el conjunto de validaci√≥n y el de entrenamiento tengan el mismo tama√±o. Esto es simplemente una forma de asegurarnos de que los resultados que obtengamos en validaci√≥n puedan ser representativos del test. Podr√≠amos hacer esta divisi√≥n de manera manual, pero hay que asegurarse de que la distribuci√≥n de las clases se mantiene lo m√°s parecida posible en todos los conjuntos, as√≠ que esta divisi√≥n tiene que hacerse de manera estrat√≠ficada. Esto quiere decir que se tendr√° en cuenta la etiqueta de cada imagen para mantener la proporci√≥n al hacer la divisi√≥n. La librer√≠a sklearn tiene implementada la funci√≥n sklearn.model_selection.train_test_split() que permite hacer esta divisi√≥n f√°cilmente:

In [ ]:
X_train, X_val, Y_train, Y_val = train_test_split(X_train, Y_train, test_size=10000, stratify=Y_train, random_state=42)

Definición del modelo¶

Una vez ya hemos inspeccionado y preparado nuestros datos, solamente nos queda definir el modelo que vamos a utilizar. Se pueden utilizar una gran cantidad de arquitecturas diferentes para un ejercicio de clasificaci√≥n, pero todas tienen que tener algo en com√∫n: la salida del modelo tiene que ser una probabilidad (clasificaci√≥n binaria, la probabilidad de la clase positiva normalmente) o una distribuci√≥n de probabilidad (clasificaci√≥n multi-clase, la probabilidad de pertenecer a cada una de las clases). A la hora de implementarlo, esto quiere decir que nuestra √∫ltima capa tiene que ser una capa densa (layers.Dense) con una o varias neuronas y funci√≥n de activaci√≥n sigmoide o softmax respectivamente.

En nuestro caso particular estamos planteando un problema de clasificacion multi-clase (tenemos que clasificar entre 10 d√≠gitos distintos), por lo que nuestra √∫ltima capa ser√° una capa densa de 10 neuronas con activaci√≥n sigmoid.

Adem√°s de la arquitectura, tendremos que elegir una funci√≥n de coste acorde: la entrop√≠a cruzada. TensorFlow implementa dos variantes de esta funci√≥n de coste: tf.losses.BinaryCrossentropy() para el caso binario, y tf.losses.CategoricalCrossentropy() (y tf.losses.SparseCategoricalCrossentropy()) para el caso multiclase. Llama la atenci√≥n que haya dos variantes para el caso multi-clase. Esto es porque las funciones cambian dependiendo de c√≥mo introduzcamos las etiquetas al modelo: si utilizamos etiquetas en formato one-hot utilizaremos tf.losses.CategoricalCrossentropy(), mientras que si nuestras etiquetas son n√∫meros discretos (0, 1, 2, 3...) utilizaremos tf.losses.SparseCategoricalCrossentropy(). Como hemos visto antes, la forma de nuestras etiquetas nos indica que debemos elegir tf.losses.SparseCategoricalCrossentropy().

De cara a la definici√≥n de los modelos, Keras permite utilizar tres aproximaciones que difieren en la sencillez y la personalizaci√≥n de los modelos:

  • API Secuencial: Es la m√°s sencilla de utilizar pero solo permite definir modelos donde las capas se conectan una detr√°s de la otra.
  • API Funcional: Permite mucha m√°s flexibilidad que la API Secuencial pero requiere que especifiquemos qu√© capas se conectan entre s√≠ y c√≥mo lo hacen.
  • Herencia de clases: Permite un control total de lo que pasa en el modelo. Solamente nos har√° falta utiliarla en casos espec√≠ficos donde no podamos utilizar las otras dos APIs. Puede dar algunos problemas a la hora de guardar y cargar los modelos.

En general, la mayor√≠a de problemas los podemos afrontar utilizando las dos primeras opciones.

API Secuencial¶

La arquitectura que vamos a plantear este ejercicio es muy sencilla, as√≠ que podemos utilizar la API Secuencial. Para ello instanciamos un objeto tf.keras.models.Sequential(), al cual le pasamos una lista con las capas que queremos que tenga el modelo. Esta clase ajusta autom√°ticamente las dimensiones de las matrices de peso de cada capa, pero necesita conocer las dimensiones de los datos de entrada. Esta informaci√≥n se la daremos mediante el par√°metro input_shape en la primera capa. El resto de capas no necesitan este par√°metro.

A modo de apunte, kernel_size como pool_size pueden tomar tanto valores √∫nicos como tuplas (d1, d2). Si introducimos un √∫nico valor d, lo tomar√° internamente como una tupla (d,d). Esto quiere decir que utilizaremos tama√±os de kernel y pooling sim√©tricos. Si queremos utilizar alguno que no sea sim√©trico tendremos que introducir directamente (d1, d2).

In [ ]:
model = tf.keras.models.Sequential([
    layers.Conv2D(filters=32, kernel_size=3, activation="relu", input_shape=(28,28,1)),
    layers.MaxPool2D(pool_size=2),
    layers.Conv2D(filters=64, kernel_size=3, activation="relu"),
    layers.MaxPool2D(pool_size=2),
    layers.Flatten(),
    layers.Dense(10, activation="softmax")
])

Una vez hemos definido la arquitectura del modelo, tendremos que compilarlo para especificar una funci√≥n de coste (que ya hemos explicado antes), un optimizador (lo est√°ndar suele ser elegir Adam) y las m√©tricas que queremos monitorizar durante el entrenamiento (precisi√≥n en este caso). Adem√°s, podemos utilizar el m√©todo .summary() para obtener un resumen del modelo que hemos creado. Tambi√©n podemos utilizar la funci√≥n tf.keras.utils.plot_model() para obtener un resumen m√°s visual, aunque requiere de la instalaci√≥n de algunas otras librer√≠as externas.

Podemos utilizar tanto las versiones "objeto" de los optimizadores y las funciones de p√©rdida como sus nombres asociados. Al utilizar los nombres no podremos cambiar par√°metros como el learning rate, pero es suficiente para este ejercicio.

Utiliza los objetos tf.optimizers.Adam() y tf.losses.SparseCategoricalCrossentropy() y cambia alguno de sus par√°metros, por ejemplo el learning rate.

Entrenamiento del modelo¶

Llegados hasta aqu√≠ solo resta utilizar el m√©todo .fit() del modelo para entrenar nuestro modelo. Este m√©todo nos permite utilizar un conjunto de validaci√≥n, validation_data, durante el entrenamiento para observar c√≥mo de bien generaliza nuestro modelo. Esto resulta muy √∫til cuando queremos utilizar t√©cnicas m√°s avanzadas como early stopping que detienen el entrenamiento cuando empezamos a sobreajustar al conjunto de datos.

Los par√°metros batch_size y epochs hacen referencia a la cantidad de im√°genes que se utilizar√°n para cada ajuste de los pesos y las veces que recorremos el conjunto de datos respectivamente. Normalmente se recomienda utilizar un batch_size lo m√°s grande posible para exprimir al m√°ximo el hardware y acelerar el entrenamiento, aunque vale la pena experimentar con distintos valores porque est√° estrechamente relacionado con el learning rate. El par√°metro verbose indica la cantidad de informaci√≥n que queremos ver durante el entrenamiento. Es interesante en modelos que se entrenan r√°pido como este, pero en modelos con entrenamientos largos o con muchas √©pocas no ser√° demasiado √∫til.

Adem√°s, el m√©todo .fit() devuelve un objeto particular que permite acceder a todas las m√©tricas que se han calculado durante el entrenamiento. Es muy √∫til para representar las din√°micas de entrenamiento del modelo.

Evaluación del modelo¶

El m√©todo .evaluate() permite calcular todas las m√©tricas que hemos compilado anteriormente en el modelo sobre el conjunto de datos que queramos. Es una buena pr√°ctica evaluar el modelo en los diferentes conjuntos de datos al acabar el entrenamiento para obtener una m√©trica final del trabajo realizado. Este m√©todo nos devuelve una lista con las m√©tricas, en este caso la funci√≥n de coste y la precisi√≥n:

Guardar el modelo¶

En este ejercicio el modelo utilizado es relativamente r√°pido de entrenar, pero en problemas reales podemos encontrar tiempos de ejecuci√≥n mucho m√°s largos. Esto quiere decir que es muy importante saber guardar el modelo una vez ha sido entrenado para poder utilizarlo sin tener que repetir el entrenamiento cada vez. Para ello podemos utilizar el m√©todo .save(). Hay varias formas posibles de guardar los modelos, aunque la m√°s sencilla es guardarlos en formato .h5:

In [ ]:
model.save("model_cnn_mnist.h5")

Una vez guardado podemos volver a cargarlo con la funci√≥n tf.keras.models.load_model() y utilizarlo normalmente:
